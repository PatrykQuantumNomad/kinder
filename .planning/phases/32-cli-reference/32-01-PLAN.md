1. **Intro paragraph** -- Explain that 4 kinder commands support `--output json` for scripting and automation. List them: `get clusters`, `get nodes`, `env`, `doctor`. Mention that JSON goes to stdout (human output goes to stderr for doctor).

2. **`## Commands`** -- One `###` subsection per command. Each subsection has:
   - The command invocation (```sh code block)
   - The JSON output shape description (one sentence)
   - An example output (```json code block)
   - 2-3 jq recipes (```sh code blocks)

Use these exact details from research:

**`### kinder get clusters`**
- Command: `kinder get clusters --output json`
- Shape: JSON array of strings (cluster names). Empty result returns `[]`.
- Example: `["kind","my-cluster"]`
- jq recipes:
  - Get first cluster: `kinder get clusters --output json | jq '.[0]'`
  - Count clusters: `kinder get clusters --output json | jq 'length'`
  - Check if cluster exists: `kinder get clusters --output json | jq 'any(. == "kind")'`

**`### kinder get nodes`**
- Command: `kinder get nodes --output json`
- Shape: JSON array of objects with `name` (string) and `role` (string: "control-plane", "worker"). Empty returns `[]`.
- Mention `--all-clusters` / `-A` flag works with `--output json`.
- Example: `[{"name":"kind-control-plane","role":"control-plane"},{"name":"kind-worker","role":"worker"}]`
- jq recipes:
  - List control-plane nodes: `kinder get nodes --output json | jq '[.[] | select(.role == "control-plane") | .name]'`
  - Count workers: `kinder get nodes --output json | jq '[.[] | select(.role == "worker")] | length'`

**`### kinder env`**
- Command: `kinder env --output json`
- Shape: Single JSON object (NOT an array). Fields: `kinderProvider`, `clusterName`, `kubeconfig`.
- Add a `:::note` callout: "kinder env outputs a single object, not an array. Use `.fieldName` to extract values, not `.[]`."
- Example: `{"kinderProvider":"docker","clusterName":"kind","kubeconfig":"/Users/user/.kube/config"}`
- Mention `--name` flag: `kinder env --name my-cluster --output json`
- jq recipes:
  - Extract provider: `kinder env --output json | jq -r '.kinderProvider'`
  - Set KUBECONFIG: `export KUBECONFIG=$(kinder env --output json | jq -r '.kubeconfig')`

**`### kinder doctor`**
- Command: `kinder doctor --output json`
- Shape: JSON array of check result objects. Fields: `name` (string), `status` ("ok"/"warn"/"fail"), `message` (string, omitted when status is "ok").
- Example all-ok: `[{"name":"docker","status":"ok"},{"name":"kubectl","status":"ok"}]`
- Example with failure: `[{"name":"container-runtime","status":"fail","message":"no container runtime found..."},{"name":"kubectl","status":"ok"}]`
- jq recipes:
  - Check if any failed: `kinder doctor --output json | jq 'any(.[]; .status == "fail")'`
  - Get failed checks: `kinder doctor --output json | jq '[.[] | select(.status == "fail") | {name, message}]'`

   - CI guard pattern: `kinder doctor --output json | jq -e 'all(.[]; .status == "ok")' > /dev/null` (exits non-zero if any check is not ok)

Structure the page as follows:

1. **Intro paragraph** -- One sentence: this page covers exit codes, check results, and common errors for `kinder env` and `kinder doctor`.

2. **`## kinder env`**

  **`### Exit codes`** -- A table:

  | Exit code | Meaning |
  |-----------|---------|
  | 0 | Success -- provider detected, output written |
  | 1 | Error -- unsupported output format or provider detection failure |

  Add a `:::note` callout: "kinder env does not verify that the cluster is running. It resolves environment variables (provider, cluster name, kubeconfig path) without contacting the container runtime."

  **`### Common errors`** -- Use the Symptom/Cause/Fix pattern (matching Phase 31 addon pages):

  Entry 1: "Unknown output format"
  - **Symptom:** `kinder env --output yaml` returns an error.
  - **Cause:** Only `json` and the default text format are supported. `yaml`, `table`, and other formats are not recognized.
  - **Fix:** Use `kinder env` (default text) or `kinder env --output json`.

  Entry 2: "Wrong cluster name in output"
  - **Symptom:** `kinder env` shows `KIND_CLUSTER_NAME=kind` but your cluster has a different name.
  - **Cause:** The `--name` flag defaults to `"kind"`. If you created a cluster with `--name my-cluster`, you must also pass `--name my-cluster` to `kinder env`.
  - **Fix:** `kinder env --name my-cluster`

  **`### Shell integration`** -- Show `eval $(kinder env)` and `eval $(kinder env --name my-cluster)` as a :::tip for setting environment variables in one step.

3. **`## kinder doctor`**

  **`### Exit codes`** -- A table:

  | Exit code | Meaning |
  |-----------|---------|
  | 0 | All checks passed |
  | 1 | One or more checks failed |
  | 2 | One or more checks warned (no failures) |

  Add: "If both failures and warnings exist, exit code 1 takes priority over exit code 2."

  **`### Check results`** -- A table explaining what each check name means:

  | Check name | What it checks | Possible statuses |
  |------------|----------------|-------------------|
  | `docker` / `podman` / `nerdctl` | Container runtime binary found and daemon responding | ok, warn, fail |
  | `container-runtime` | Fallback name when no runtime binary is found at all | fail |
  | `kubectl` | kubectl binary found and `kubectl version --client` succeeds | ok, warn |

  Add a `:::note` callout: "The check name is the actual runtime found (e.g., `docker`), not a generic label. The name `container-runtime` only appears when no runtime binary is found."

  Explain the detection order: docker, then podman, then nerdctl. Stops at the first found binary.

  **`### Common scenarios`** -- Use Symptom/Cause/Fix pattern for 3-4 entries:

  Entry 1: "No container runtime found"
  - **Symptom:** Exit code 1. JSON output shows `{"name":"container-runtime","status":"fail","message":"no container runtime found..."}`.
  - **Cause:** None of `docker`, `podman`, or `nerdctl` is installed or in PATH.
  - **Fix:** Install Docker (https://docs.docker.com/get-docker/), Podman (https://podman.io/getting-started/installation), or nerdctl (https://github.com/containerd/nerdctl). Then re-run `kinder doctor`.

  Entry 2: "Docker found but not responding"
  - **Symptom:** Exit code 2. JSON shows `{"name":"docker","status":"warn","message":"docker found but not responding..."}`.
  - **Cause:** The `docker` binary is on PATH but the Docker daemon is not running.
  - **Fix:** Start the Docker daemon (`systemctl start docker` on Linux, open Docker Desktop on macOS/Windows), then re-run `kinder doctor`.

  Entry 3: "kubectl not found"
  - **Symptom:** Exit code 1 (if combined with runtime fail) or exit code 2 (if runtime is ok but kubectl warns). JSON shows `{"name":"kubectl","status":"warn"}` or `{"name":"kubectl","status":"fail"}`.
  - **Cause:** `kubectl` is not installed or not in PATH.
  - **Fix:** Install kubectl: https://kubernetes.io/docs/tasks/tools/. Then re-run `kinder doctor`.

  Entry 4: "All checks pass but cluster still fails"
  - **Symptom:** `kinder doctor` returns exit code 0 but `kinder create cluster` fails.
  - **Cause:** `kinder doctor` checks prerequisites (runtime + kubectl), not cluster state. Issues like insufficient disk space, port conflicts, or Docker resource limits are not checked.
  - **Fix:** Check Docker resource settings (memory, disk), ensure no port conflicts on the host, and review the `kinder create cluster` error output.

  **`### Using doctor in CI`** -- A :::tip callout showing a CI guard pattern:
  ```sh
  kinder doctor --output json | jq -e 'all(.[]; .status == "ok")' > /dev/null
  if [ $? -ne 0 ]; then
    echo "Prerequisites not met"
    kinder doctor
    exit 1
  fi
  ```

Do NOT claim kinder env has exit code 2. Do NOT claim kinder env checks if the cluster is running. Follow the Symptom/Cause/Fix pattern from Phase 31 addon pages. Do NOT touch astro.config.mjs or create new files.
