    **Page 1: `addons/metallb.md` (DOCS-04)**

    Frontmatter: `title: MetalLB`, `description: MetalLB addon for LoadBalancer service support in kinder clusters.`

    Content:
    - **Overview**: kinder installs MetalLB v0.15.3 to provide LoadBalancer service support. Without MetalLB, Services of type LoadBalancer in kind/kinder clusters stay in Pending state forever because there is no cloud provider to allocate external IPs.
    - **What gets installed**: MetalLB deployed to `metallb-system` namespace. An `IPAddressPool` named `kind-pool` and an `L2Advertisement` named `kind-l2advert` are created automatically.
    - **IP address pool**: kinder auto-detects the Docker/Podman network subnet used by the kind cluster and carves out a pool from the `.200-.250` range of the last octet (e.g., if the network is 172.18.0.0/16, the pool is 172.18.255.200-172.18.255.250).
    - **How to verify**: `kubectl get pods -n metallb-system` (controller and speaker pods Running), then create a LoadBalancer service and check `kubectl get svc` shows an EXTERNAL-IP from the pool.
    - **Configuration**: The config field is `addons.metalLB`. Link to the configuration reference page.
    - **How to disable**: Show YAML snippet:
      ```yaml title="kinder-config.yaml"
      kind: Cluster
      apiVersion: kind.x-k8s.io/v1alpha4
      addons:
        metalLB: false
      ```
    - **Platform notes**: Use `:::caution` aside: "MetalLB's L2 speaker cannot send ARP in rootless Podman mode. If you are using rootless Podman, LoadBalancer services may not receive an EXTERNAL-IP. This is a known limitation of L2 mode in network-restricted environments."

    **Page 2: `addons/envoy-gateway.md` (DOCS-05)**

    Frontmatter: `title: Envoy Gateway`, `description: Envoy Gateway addon for Gateway API routing in kinder clusters.`

    Content:
    - **Overview**: kinder installs Envoy Gateway v1.3.1 to provide Gateway API support. This gives you the full Gateway API CRD set (GatewayClass, Gateway, HTTPRoute, etc.) and a working Envoy-based data plane for HTTP/HTTPS routing.
    - **What gets installed**: Envoy Gateway deployed to `envoy-gateway-system` namespace. A `GatewayClass` named `eg` is created with controller `gateway.envoyproxy.io/gatewayclass-controller`.
    - **How to use**: Show a minimal Gateway + HTTPRoute example that references GatewayClass `eg`:
      ```yaml
      apiVersion: gateway.networking.k8s.io/v1
      kind: Gateway
      metadata:
        name: my-gateway
      spec:
        gatewayClassName: eg
        listeners:
          - name: http
            protocol: HTTP
            port: 80
      ```
    - **How to verify**: `kubectl get gatewayclass eg` (ACCEPTED=True), `kubectl get pods -n envoy-gateway-system` (pods Running).
    - **Configuration**: The config field is `addons.envoyGateway`. Link to the configuration reference page.
    - **How to disable**: Show YAML snippet with `envoyGateway: false`.
    - **Technical note**: Use `:::note` aside: "kinder uses `kubectl apply --server-side` to install Envoy Gateway because the Gateway API CRDs (particularly HTTPRoutes) exceed the 256 KB annotation limit for standard client-side apply. If you ever need to manually re-apply Envoy Gateway manifests, use `kubectl apply --server-side -f <manifest>`."

    **Page 3: `addons/metrics-server.md` (DOCS-06)**

    Frontmatter: `title: Metrics Server`, `description: Metrics Server addon for resource metrics and autoscaling in kinder clusters.`

    Content:
    - **Overview**: kinder installs Metrics Server v0.8.1 to enable resource metrics collection. This makes `kubectl top` commands work and enables Horizontal Pod Autoscaler (HPA) functionality.
    - **What gets installed**: Metrics Server deployed as `deployment/metrics-server` in the `kube-system` namespace. It runs with `--kubelet-insecure-tls` to work with kind's self-signed kubelet certificates.
    - **What you get**: `kubectl top nodes` shows CPU and memory usage per node. `kubectl top pods` shows resource usage per pod. HPA resources can scale workloads based on CPU/memory metrics.
    - **How to verify**: `kubectl top nodes` (shows CPU and memory columns with values, not `<unknown>`).
    - **Configuration**: The config field is `addons.metricsServer`. Link to the configuration reference page.
    - **How to disable**: Show YAML snippet with `metricsServer: false`.
    - Use `:::tip` aside: "Metrics may take 30-60 seconds to become available after cluster creation. If `kubectl top nodes` shows `error: metrics not available yet`, wait a moment and try again."

    **Page 4: `addons/coredns.md` (DOCS-07)**

    Frontmatter: `title: CoreDNS Tuning`, `description: CoreDNS performance tuning applied by kinder to improve DNS resolution.`

    Content:
    - **Overview**: kinder applies three performance tuning patches to the default CoreDNS configuration in the `kube-system` namespace. Unlike other addons, this does not install new software — it modifies the existing CoreDNS deployment that Kubernetes ships by default.
    - **What changes**: Show a before/after Corefile comparison:
      1. **autopath @kubernetes** inserted before the `kubernetes` plugin block — enables automatic DNS search path optimization so pods can resolve short names (e.g., `my-service` instead of `my-service.default.svc.cluster.local`) with fewer DNS round-trips.
      2. **pods insecure → pods verified** — enables security validation for pod DNS records, verifying that A records for pods match actual pod IPs.
      3. **cache 30 → cache 60** — doubles the DNS cache TTL from 30 to 60 seconds, reducing DNS query load on CoreDNS.
    - Show the before/after Corefile blocks:
      ```text title="Before (default)"
      .:53 {
          kubernetes cluster.local ...
          pods insecure
          cache 30
      }
      ```
      ```text title="After (kinder)"
      .:53 {
          autopath @kubernetes
          kubernetes cluster.local ...
          pods verified
          cache 60
      }
      ```
    - **How to verify**: `kubectl get configmap coredns -n kube-system -o yaml` and check that the Corefile contains `autopath @kubernetes`, `pods verified`, and `cache 60`.
    - **Configuration**: The config field is `addons.coreDNSTuning`. Link to the configuration reference page.
    - **How to disable**: Show YAML snippet with `coreDNSTuning: false`.

    **Page 5: `addons/headlamp.md` (DOCS-08)**

    Frontmatter: `title: Headlamp Dashboard`, `description: Headlamp web dashboard addon for visual Kubernetes cluster management.`

    Content:
    - **Overview**: kinder installs Headlamp v0.40.1, a web-based Kubernetes dashboard, in the `kube-system` namespace. Headlamp provides a visual interface for viewing and managing cluster resources.
    - **What gets installed**: Headlamp deployed as `deployment/headlamp` in `kube-system`. A service account with a long-lived token secret named `kinder-dashboard-token` is created for authentication.
    - **Accessing the dashboard**: Show the port-forward command and URL:
      ```bash
      kubectl port-forward -n kube-system service/headlamp 8080:80
      ```
      Then open http://localhost:8080 in your browser. Use the token printed during `kinder create cluster` to log in.
    - **Retrieving the token**: Use `:::tip` aside: "kinder prints the dashboard token once during cluster creation. If you missed it or need to retrieve it later, run:"
      ```bash
      kubectl get secret kinder-dashboard-token -n kube-system \
        -o jsonpath='{.data.token}' | base64 -d
      ```
    - **How to verify**: `kubectl get pods -n kube-system -l app.kubernetes.io/name=headlamp` (pod Running), then access via port-forward.
    - **Configuration**: The config field is `addons.dashboard`. Link to the configuration reference page.
    - **How to disable**: Show YAML snippet with `dashboard: false`.
