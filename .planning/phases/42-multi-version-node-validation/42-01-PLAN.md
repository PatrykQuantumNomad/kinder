Output: ExplicitImage sentinel on internal Node type, fixed fixupOptions, validateVersionSkew in Cluster.Validate(), comprehensive tests.

1. In `pkg/internal/apis/config/types.go`, add `ExplicitImage bool` field to the `Node` struct after the `Image` field. Add a comment: "ExplicitImage is true when the user explicitly set image in config. When true, the --image flag will not override this node's image."

2. In `pkg/internal/apis/config/zz_generated.deepcopy.go`, the `DeepCopyInto` for `Node` uses `*out = *in` for value types which automatically copies bool fields, so verify no manual change is needed. If the Node deepcopy does field-by-field copying, add `out.ExplicitImage = in.ExplicitImage`.

3. In `pkg/internal/apis/config/convert_v1alpha4.go`, in `convertv1alpha4Node`, after `out.Image = in.Image`, add: `out.ExplicitImage = in.Image != ""`. This detects whether the user explicitly specified an image in their config. Note: `SetDefaultsNode` is called before conversion and fills in `defaults.Image` when Image is empty, so by the time we convert, `in.Image` is always non-empty. The correct approach is to check BEFORE defaults are applied. Since `V1Alpha4ToInternal` calls `SetDefaultsCluster` (which calls `SetDefaultsNode`) BEFORE `Convertv1alpha4`, we need a different sentinel. The cleanest fix: check if the v1alpha4 Node's Image differs from `defaults.Image`. But this is fragile. Better approach: In `convert.go` (`V1Alpha4ToInternal`), record which nodes had non-empty Image BEFORE calling `SetDefaultsCluster`, then set ExplicitImage after conversion. Specifically:
   - In `encoding/convert.go`, modify `V1Alpha4ToInternal`: before calling `v1alpha4.SetDefaultsCluster(cluster)`, iterate over `cluster.Nodes` and record a `[]bool` of which nodes have `Image != ""`. After conversion, set `out.Nodes[i].ExplicitImage = explicitFlags[i]` for each node.

4. In `pkg/cluster/internal/create/create.go`, in `fixupOptions`, change the image override loop (lines 379-385) from unconditional override to conditional:
   ```go
   if opts.NodeImage != "" {
       for i := range opts.Config.Nodes {
           if !opts.Config.Nodes[i].ExplicitImage {
               opts.Config.Nodes[i].Image = opts.NodeImage
           }
       }
   }
   ```

5. Write tests in `pkg/internal/apis/config/validate_test.go` (or a new `fixup_test.go` in the create package) covering the behavior cases above.

    - Test: single control-plane v1.31.2 + two workers at v1.26.0 and v1.27.1 -> error reporting BOTH violations in one table
    - Test: HA control-plane with same minor version (v1.31.0 and v1.31.2) -> no error

1. In `pkg/internal/apis/config/validate.go`, add a helper function `imageTagVersion(image string) (string, error)` that extracts the version tag from a node image string. Strip any `@sha256:` digest suffix first, then extract the substring after the last `:`. Return error if no tag is found.

2. Add a `skewViolation` struct:
   ```go
   type skewViolation struct {
       NodeName string // role[index] format, e.g. "worker[1]"
       Role     string
       Version  string
       Delta    int    // negative means behind
   }
   ```

3. Add `validateVersionSkew(nodes []Node) error` function:
   a. Extract version from each node's Image using `imageTagVersion`. If any image has no parseable tag, return an error identifying the node.
   b. Parse each version string with `version.ParseSemantic`.
   c. Find control-plane nodes. If multiple control-plane nodes exist, verify they all share the same minor version. If not, collect this as an HA mismatch error with reason: "HA control-plane nodes must run the same minor version for etcd consistency; found v1.X and v1.Y".
   d. Using the control-plane minor version as reference, check each worker node. If any worker's minor version is more than 3 behind the control-plane minor, record a `skewViolation`.
   e. If violations exist, format a table-style error message matching the user's specified format:
      ```
      Error: version-skew policy violated

        Node        Role           Version  Delta
        worker[1]   worker         v1.27.1  -4
        worker[2]   worker         v1.26.0  -5

      Control-plane version: v1.31.2
      Max allowed skew: 3 minor versions

      Hint: update worker[1] to v1.28+ and worker[2] to v1.28+
      ```
   f. Use `fmt.Sprintf` to build the error string, NOT tabwriter (keep it simple for error messages). Use fixed-width formatting with `%-12s` style.
   g. Return the formatted error via `errors.Errorf(...)`.

4. In `Cluster.Validate()`, call `validateVersionSkew(c.Nodes)` AFTER the existing node validation loop (after individual nodes are validated). Append any error to `errs`.

5. Add comprehensive tests in `validate_test.go` covering all behavior cases above. Follow the existing `TestClusterValidate` table-driven pattern. For test nodes, use images like `kindest/node:v1.31.2`, `kindest/node:v1.27.1`, etc. Ensure ExplicitImage is set appropriately on test nodes (true for all nodes with explicit images in multi-version scenarios).
  </action>
