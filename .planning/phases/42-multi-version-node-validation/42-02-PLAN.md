1. Create `pkg/internal/doctor/clusterskew.go` implementing a `clusterNodeSkewCheck` struct.

2. The check needs to interact with a running cluster. Since doctor checks currently don't take provider/cluster arguments (they use `Run() []Result` with no params), use dependency injection similar to other checks. The check struct should have injectable functions:
   ```go
   type clusterNodeSkewCheck struct {
       listNodes   func() ([]nodeEntry, error)  // abstracts provider.ListNodes + KubeVersion + image inspect
   }

   type nodeEntry struct {
       Name    string
       Role    string
       Version string  // raw version string from /kind/version (live kubelet version)
       Image   string  // container image tag, e.g. "kindest/node:v1.31.2" (from container inspect)
   }
   ```

3. The `newClusterNodeSkewCheck()` constructor creates a real implementation that:
   a. Creates a provider via `cluster.NewProvider()` with a no-op logger
   b. Lists nodes for the default cluster (`cluster.DefaultName`)
   c. For each node, calls `nodeutils.KubeVersion(n)` to get the live version from `/kind/version`
   d. For each node, calls `n.Role()` to get the role
   e. For each node, retrieves the container image using the same inspect pattern as `Role()`. Since `nodes.Node` doesn't expose an Image() method directly, and the doctor check needs the container image tag, use `n.Command("cat", "/kind/image")` if that file exists, OR reconstruct the image as `kindest/node:<version>` from KubeVersion output as a fallback. The most reliable approach: the `Node` interface embeds `exec.Cmder`, so the check can run a command inside the container. However, the container image is not stored inside the container -- it's metadata on the container runtime. Since the doctor check doesn't have access to the provider's binary name for `docker inspect`, use the pragmatic approach: extract the image tag version from the node name pattern or reconstruct from KubeVersion. For config drift detection, the key comparison is: does the version baked into the image tag match the live `/kind/version`? Since both come from the same image in normal operation, drift would indicate the `/kind/version` file was modified post-creation. The practical implementation:
      - Set `Image` on `nodeEntry` to `"kindest/node:" + version` (reconstructed from KubeVersion)
      - This means in normal operation there's no drift (image tag version == /kind/version by construction)
      - Config drift is only detectable if we can independently read the container image. Since we cannot reliably access `docker inspect` from inside the doctor check (no BinaryName access), document this limitation in the code and implement drift detection as: compare the image tag version (if available) against the live version. For the initial implementation, the check will reconstruct Image from Version, making drift undetectable in practice.

   **REVISED approach for config drift:** Since the `nodes.Node` interface doesn't expose the container image and the doctor check has no access to the container runtime binary, config drift between "image used to create the container" and "live /kind/version" cannot be reliably detected without extending the Node interface (out of scope for this phase). Instead:
   - Implement the skew policy check fully (this IS detectable from /kind/version alone)
   - For the Image field on nodeEntry, reconstruct as `kindest/node:<version>` from KubeVersion output
   - Add a code comment documenting that config drift detection would require adding an `Image()` method to the `nodes.Node` interface, which is a broader change deferred to a future phase
   - Remove "config drift" from the check's scope -- the doctor check validates upstream skew policy on running clusters

4. `Run()` logic:
   a. Call `listNodes()`. If error or empty, return a skip result.
   b. Parse each node's version with `version.ParseSemantic`.
   c. Find control-plane nodes and their minor version.
   d. Check HA control-plane same-minor-version requirement.
   e. Check workers within 3 minor versions of control-plane.
   f. If no violations, return `ok` with message listing all node versions.
   g. If violations found, return `warn` with:
      - Same table format as create-time validation (per user decision):

        ```
        Node        Role           Version  Delta
        worker-2    worker         v1.27.1  -4
        ```
      - Reason: "cluster node version-skew exceeds Kubernetes policy (max 3 minor versions)"
      - Fix: "update <node> to v1.28+ ..." (remediation hints matching create-time format)
   h. Severity is `warn` (per Claude's Discretion decision, consistent with kubectlVersionSkewCheck).
