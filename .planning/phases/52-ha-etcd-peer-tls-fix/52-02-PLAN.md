<objective>
Implement the IP-pin path: at HA cluster create time, record each control-plane
container's IP, persist it inside the container at `/kind/ipam-state.json`,
reconnect the container to the kind network with `--ip <recorded-ip>`, and
label the container with the resume-strategy. This is the preferred path per
roadmap and CONTEXT.md.

Per CONTEXT.md locked decisions:
- IP pinning is the preferred fix; cert-regen is the fallback (Plan 52-03).
- Single-CP clusters incur zero overhead.
- **Support all 3 runtimes** (docker, podman, nerdctl) — each independently
  gated by the probe verdict. Docker AND Podman get IP-pin wiring (both
  PASS the probe per RESEARCH "Probe: Concrete Definition" — Docker
  HIGH confidence, Podman MEDIUM confidence with `--ip` flag verified).
  Nerdctl gets NO wiring because its probe verdict is hard-coded
  `VerdictUnsupported` (Plan 52-01) and `nerdctl network connect` does
  not exist (RESEARCH PIT-4) — its containers fall through to no-label,
  which Plan 52-03 treats as legacy = cert-regen.
- No migration command for existing clusters (Plan 52-02 ONLY operates at
  fresh cluster create time).

Per RESEARCH (HIGH confidence):
- Storage: `/kind/ipam-state.json` mirrors the `/kind/pause-snapshot.json`
  precedent (Phase 47 pattern).
- The container is RUNNING at the post-Provision hook point, so `docker exec`
  is the legal write path (PIT-2 gates exec on running state — running here is
  fine).
- Labels can ONLY be set at `docker run` time (no `docker container update
  --label-add` exists). The IP is unknown at run time. Therefore the label
  marks STRATEGY (`ip-pinned` vs `cert-regen` vs absent=legacy), and the IP
  itself lives in the JSON file.
- Podman parity: `podman` provider has the same `Provision(...) →
  ensureNetwork → planCreation → UntilErrorConcurrent` shape as docker
  (verified at `pkg/cluster/internal/providers/podman/provider.go`). The
  podman wiring mirrors the docker wiring change.

Important constraint surfaced by RESEARCH and Pitfall 1: this plan does NOT
operate on stopped containers, does NOT touch resume.go, and does NOT regen
certs. It only writes state and pins IPs at create time. Plan 52-03 owns the
resume-time orchestration that consumes what this plan persists.

Purpose: Deliver the recording side of the IP-pin contract for ALL pinning-
capable runtimes (docker + podman). Plan 52-03 will deliver the consuming
side (resume-time disconnect+reconnect with the recorded IP) and the
cert-regen fallback.
