<objective>
Plan 52-03 owns the resume-time orchestration that consumes Plan 52-02's
recorded state. It delivers BOTH paths:

1. The IP-pin RESUME hook: read `/kind/ipam-state.json` for each CP via
   `docker cp` (containers are stopped at this point — exec is forbidden,
   cp works), then disconnect + reconnect with `--ip <recorded>` BEFORE the
   existing `docker start` call. After this, `docker start` finds the
   container already at the pinned IP, etcd peer SANs match, etcd recovers
   cleanly. The network name used for disconnect/connect comes from the
   `Network` field of the JSON state file — already persisted by Plan 52-02
   — so resume does not need to re-derive networkName from a possibly-
   ambiguous `docker inspect` (a CP container attached to multiple networks
   would yield an ambiguous result; the JSON file removes that ambiguity).

2. The CERT-REGEN fallback: for clusters labeled `cert-regen` or with NO
   label (legacy, per CONTEXT.md "cert-regen forever for legacy" Claude's
   Discretion), reactively detect IP drift after all containers are started,
   then run wholesale `kubeadm certs renew etcd-peer` on every CP node and
   cycle the etcd static pod.

Per CONTEXT.md locked decisions:
- All 3 runtimes covered.
- Cert-regen happens AFTER container start because kubeadm needs the API
  server reachable (RESEARCH PIT-5). IP-pin happens BEFORE container start
  because network mutation requires stopped state (RESEARCH PIT-1).
- Doctor stays quiet on legacy clusters (no warning emitted just for being
  legacy — the new doctor check from Plan 52-04 surfaces strategy on demand
  only).

Per CONTEXT.md Claude's Discretion (locked recommendations from RESEARCH):
- Trigger: REACTIVE — only on IP change.
- Scope: WHOLESALE — all CP nodes.
- Failure recovery: HALT + DIAGNOSTIC.
- Progress UX: single-line spinner per CP, matching `cli.StatusForLogger`.
- Legacy detection: absence of resume-strategy label.

Per W2 disposition (Option A — locked here, not Option B):
- The cluster-level resume-strategy label is read ONCE from the bootstrap
  CP. If it is `ip-pinned`, the cluster's contract is that EVERY CP carries
  a valid `/kind/ipam-state.json`. Any `ReadIPAMState` failure on any CP
  during the pre-start phase is therefore a CLUSTER CORRUPTION SIGNAL, not
  a per-CP legacy condition. Resume HALTS with a structured diagnostic
  error directing the user to delete and recreate the cluster. There is
  NO soft-skip and NO per-CP fallback to cert-regen mid-resume — that
  would mask cluster damage and produce undefined cluster state. Option B
  (soft-skip + per-cluster fallback) was REJECTED because it requires
  additional integration tests, adds undefined intermediate states, and
  contradicts the "halt + diagnostic" pattern locked elsewhere in this
  phase. Option A is consistent with the disconnect-failure halt behavior
  in Plan 52-02 Task 1.

Per RESEARCH (HIGH/MEDIUM confidence):
- `kubeadm certs renew etcd-peer` is the correct kubeadm command.
- Static pod cycle: move manifest out, sleep 25s (kubelet
  fileCheckFrequency=20s + 5s margin), move back, sleep 20s.
- `docker cp` works on stopped containers; `docker exec` does NOT.
- Etcd cannot tolerate mixed-cert state across peers — must regen wholesale.
- Etcd peer cert SAN list uses container IPs (RESEARCH Open Question 3 —
  RESOLVED with MEDIUM confidence; live verification recorded as Manual
