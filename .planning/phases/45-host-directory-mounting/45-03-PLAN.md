    Create `kinder-site/src/content/docs/guides/host-directory-mounting.md` following the same structure as `local-dev-workflow.md` and `hpa-auto-scaling.md`.

    **Frontmatter:**

    ```yaml
    ---
    title: Host Directory Mounting
    description: Mount host directories into cluster nodes and expose them to pods via hostPath PersistentVolumes.
    ---
    ```

    **Content structure (follow the pattern of existing guides):**

    1. **Opening paragraph** — Explain the two-hop pattern in one paragraph: kinder nodes are containers, so mounting a host directory into a pod requires two steps: first mount the host directory into the node container (via `extraMounts` in the cluster config), then expose the node path to pods via a `hostPath` PersistentVolume. This guide walks through the complete setup.

    2. **Prerequisites section:**
       - kinder installed
       - Docker (or Podman) installed and running
       - `kubectl` installed and on PATH
       - A directory on the host you want to share (e.g., `~/shared-data`)

    3. **Step 1: Create the host directory**
       ```sh
       mkdir -p ~/shared-data
       echo "hello from host" > ~/shared-data/greeting.txt
       ```

    4. **Step 2: Create the cluster with extraMounts**
       - Show a cluster config YAML file (`mount-cluster.yaml`):
       ```yaml
       apiVersion: kind.x-k8s.io/v1alpha4
       kind: Cluster
       nodes:
         - role: control-plane
           extraMounts:
             - hostPath: /Users/you/shared-data
               containerPath: /mnt/host-data
       ```
       - Explain: `hostPath` is the absolute path on the host machine, `containerPath` is where it appears inside the node container.
       - Command: `kinder create cluster --config mount-cluster.yaml`

       Add a tip (using Astro's `:::tip` syntax): "Replace `/Users/you/shared-data` with the absolute path to your directory. On Linux, any path works. On macOS, the path must be within Docker Desktop's file-sharing directories (see Platform Notes below)."

    5. **Step 3: Create PersistentVolume and PersistentVolumeClaim**
       - Explain: The directory is now inside the node container at `/mnt/host-data`. To make it available to pods, create a `hostPath` PV pointing to that path, and a PVC to claim it.
       - Show the YAML (as a single multi-document YAML block):
       ```yaml
       apiVersion: v1
       kind: PersistentVolume
       metadata:
         name: host-data-pv
       spec:
         capacity:
           storage: 10Gi
         accessModes:
           - ReadWriteOnce
         hostPath:
           path: /mnt/host-data
         storageClassName: manual
       ---
       apiVersion: v1
       kind: PersistentVolumeClaim
       metadata:
         name: host-data-pvc
       spec:
         accessModes:
           - ReadWriteOnce
         resources:
           requests:
             storage: 10Gi
         storageClassName: manual
       ```
       - Note: `storageClassName: manual` avoids conflict with the default `local-path` StorageClass. The PV `hostPath.path` matches the `containerPath` from the cluster config.
       - Command: `kubectl apply -f pv.yaml`

    6. **Step 4: Deploy a pod that uses the mount**
       - Show a pod YAML:
       ```yaml
       apiVersion: v1
       kind: Pod
       metadata:
         name: data-reader
       spec:
         containers:
           - name: reader
             image: busybox:1.37.0
             command: ["sh", "-c", "cat /data/greeting.txt && sleep 3600"]
             volumeMounts:
               - name: host-data
                 mountPath: /data
         volumes:
           - name: host-data
             persistentVolumeClaim:
               claimName: host-data-pvc
       ```
       - Command: `kubectl apply -f pod.yaml`

    7. **Step 5: Verify the mount**
       ```sh
       kubectl logs data-reader
       ```
       - Expected output: `hello from host`
       - Explain: The file created on the host in Step 1 is visible inside the pod. Changes made on the host appear in the pod, and vice versa (both directions, since the mount is read-write).

    8. **How it works** section — A short explanation with a visual flow:
       ```
       Host directory          Node container         Pod
       ~/shared-data    --->   /mnt/host-data   --->  /data
       (extraMounts)           (hostPath PV)          (volumeMount)
       ```
       Explain: The first hop is the Docker bind mount (`extraMounts`), the second hop is the Kubernetes `hostPath` volume. Because kinder nodes are containers (not VMs), the `hostPath` PV points to a path inside the node container, not directly to the host filesystem.

    9. **Platform Notes** section:

       **macOS (Docker Desktop):**
       - Docker Desktop only shares specific host directories with containers. By default: `/Users`, `/Volumes`, `/private`, `/tmp`, `/var/folders`.
       - If your host path is outside these directories, add it in Docker Desktop Settings -> Resources -> File Sharing.
       - Mount propagation modes (`HostToContainer`, `Bidirectional`) are not supported on Docker Desktop and will be silently treated as `None`. kinder emits a warning if you specify a non-None propagation mode.
       - Run `kinder doctor` to check if your paths are correctly configured.

       **Windows (Docker Desktop):**
       - Similar file-sharing restrictions apply. Check Docker Desktop settings.
       - Mount propagation is not supported.

       **Linux:**
       - All host paths are accessible. Mount propagation works as expected.

    10. **Troubleshooting** section with `:::caution` blocks:

        - **"host path does not exist" error at cluster creation** — kinder validates that all `extraMounts` host paths exist before creating any containers. Create the directory first: `mkdir -p /path/to/dir`.
        - **Pod stuck in Pending** — Check that the PV `hostPath.path` matches the `containerPath` from your cluster config, not the host path.
        - **Empty directory in pod** — Verify the host directory has files. On macOS, verify Docker Desktop file sharing is enabled for the host path.

    11. **Cleanup** section:
        ```sh
        kubectl delete pod data-reader
        kubectl delete pvc host-data-pvc
        kubectl delete pv host-data-pv
        kinder delete cluster
        ```

    Use Astro Starlight admonition syntax (`:::tip`, `:::note`, `:::caution`) matching the pattern in existing guides. Do NOT use HTML or custom components.
