<objective>
Replace the unreachable `which etcdctl` probe in `cluster-resume-readiness` (plan 47-04) and `readEtcdLeaderID` (plan 47-02) with a `crictl exec`-based probe that talks to the etcd static-pod container directly. The current implementations probe the kindest/node rootfs for `etcdctl`, which never succeeds because etcd runs as a static pod and ships `etcdctl` only inside the `registry.k8s.io/etcd:VERSION` container's filesystem. As a result, the doctor check ALWAYS reports `skip` with "etcdctl unavailable inside container" on every real HA kinder cluster — Phase 47 success criterion 4 ("warn if quorum at risk") is never delivered in production.

Purpose: Close the gap surfaced by manual verification on a real 3-CP HA cluster (see 47-VERIFICATION.md human_needed scenario #4). Make `cluster-resume-readiness` actually warn on real etcd quorum loss. As a side fix, make the pause-time leader-ID snapshot capture work too — both probes share the same broken path.

Output: Working HA quorum probe on production kinder clusters, validated by unit tests for the new crictl invocation shape AND a real-cluster smoke test command in the verification block.

Probe redesign: **Option B from gap_summary — `crictl exec <etcd-container-id> etcdctl ...`**

Rationale (documented for executor):
- **Option A (kubectl exec via static pod)** rejected: requires the Kubernetes API server to be fully responsive AND the etcd static pod to be Ready. The check fires precisely when quorum may be at risk — when the API server may be unreachable. Self-defeating.
- **Option C (curl to /health)** rejected: per-member endpoint, would need to enumerate members from somewhere (back to needing etcdctl), AND `/health` doesn't expose leader id, so the snapshot freshness comparison would die. Loses behavior.
- **Option B (crictl exec)** chosen: crictl IS on kindest/node (verified — used at `pkg/cluster/provider.go:332`, `pkg/cluster/nodeutils/util.go:203,223`). Independent of Kubernetes API health. Talks straight to containerd. Preserves both `etcdctl endpoint health` AND `etcdctl endpoint status` invocations so the leader rotation detection from plan 47-04 survives. Zero new module dependencies (crictl is a runtime tool, not a Go import).
</objective>

<crictl_usage_pattern>
<!-- The executor needs to know what crictl exec looks like for the etcd static pod. -->
<!-- Verified empirically; not assumed. -->

The etcd static pod creates a container labeled with the pod name "etcd-<bootstrap-cp-name>".
crictl filters by container name (NOT pod name) using `--name etcd`. This matches container
name, which kubelet sets to the container's name field in the static-pod manifest — for the
kube-system/etcd-* static pod that's literally "etcd".

Discovery (returns one container id per running etcd container):
  crictl ps --name etcd -q
  → e.g. "abc123def456" (just the id, one per line)

Exec (run etcdctl inside the etcd container's filesystem):
  crictl exec <container-id> etcdctl --cacert=... --cert=... --key=... \
      --endpoints=https://127.0.0.1:2379 endpoint health --cluster --write-out=json

Cert paths INSIDE the etcd container are the same as inside the kindest/node rootfs
(/etc/kubernetes/pki/etcd/) because kubelet bind-mounts /etc/kubernetes/pki into the
etcd container per the static-pod manifest. The same etcdctlAuthArgs from the existing
code apply unchanged.

Skip path:
  - `crictl ps --name etcd -q` errors → skip "crictl unavailable inside container"
  - `crictl ps --name etcd -q` returns empty (no running etcd container) → skip
    "etcd container not running"
  - `crictl exec ... etcdctl` errors with non-zero exit → handled as
    "etcd endpoint health probe failed" (warn) — same as the current code
</crictl_usage_pattern>
