    ID         string
    Containers map[string]map[string]string
}

        if len(networks[i].Containers) > len(networks[j].Containers) {

        }
        return networks[i].ID < networks[j].ID  // bug: only correct when container counts are equal
    })
}

    In network_test.go, add test cases to the existing `Test_sortNetworkInspectEntries` function. Add these cases that expose the bug:

    1. "equal container count tiebreaker" — Two networks both with 1 container, different IDs. Currently the sort is CORRECT for this case (falls through to ID comparison). This is a regression guard.

    2. "more containers wins over lower ID" — Network A has 0 containers with ID "aaa...", Network B has 1 container with ID "zzz...". Expected sort: B first (more containers), then A. The BUGGY comparator would put A first because `networks[i].ID < networks[j].ID` is checked even when container counts differ. Specifically:
       - When i=A, j=B: `len(A.Containers)=0 > len(B.Containers)=1` is false, falls through to `A.ID < B.ID` which is true. So the comparator says A < B (A should come first), which is WRONG — B has more containers and should come first.

    **Apply fix (GREEN):**

    In network.go, replace the sort comparator (lines 205-211):

    FROM:
    ```go
    sort.Slice(networks, func(i, j int) bool {
        if len(networks[i].Containers) > len(networks[j].Containers) {
            return true
        }
        return networks[i].ID < networks[j].ID
    })
    ```

    TO:
    ```go
    sort.Slice(networks, func(i, j int) bool {
        iLen := len(networks[i].Containers)
        jLen := len(networks[j].Containers)
        if iLen != jLen {
            return iLen > jLen
        }
        return networks[i].ID < networks[j].ID
    })
    ```

    The fix: Only fall through to ID comparison when container counts are EQUAL (`iLen != jLen` guard). When counts differ, the network with more containers always wins. This satisfies strict weak ordering — for equal elements, the comparator returns false (not less-than).
