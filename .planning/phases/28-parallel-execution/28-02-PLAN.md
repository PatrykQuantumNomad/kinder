Define this as a package-level function (not a closure) after the `AddonEntry` struct:
```go
// runAddonTimed executes a single addon action and returns the result with timing.
// Disabled addons are skipped immediately (zero duration).
func runAddonTimed(actionsCtx *actions.ActionContext, name string, enabled bool, a actions.Action) addonResult {
    if !enabled {
        actionsCtx.Logger.V(0).Infof(" * Skipping %s (disabled in config)\n", name)
        return addonResult{name: name, enabled: false}
    }
    start := time.Now()
    err := a.Execute(actionsCtx)
    dur := time.Since(start)
    if err != nil {
        actionsCtx.Logger.Warnf("Addon %s failed to install (cluster still usable): %v", name, err)
    }
    return addonResult{name: name, enabled: true, err: err, duration: dur}
}
```

**7. cli.Status concurrency handling:**

Inspecting `pkg/internal/cli/status.go`: the `Status.Start()` method calls `s.End(true)` then sets `s.status = status`. The `s.status` field is NOT protected by any mutex. Two concurrent goroutines calling `Start()` will race on `s.status`.

However, the `Spinner` itself (the underlying writer) IS internally mutex-protected. The data race is specifically on `Status.status` (a string field) and `Status.spinner.SetSuffix/Start/Stop`.

The safest approach without modifying cli.Status (which is shared infrastructure from kind): **Do NOT modify Status or Spinner.** Instead, rely on the fact that `actionsContext` is shared as a pointer and `Status.Start/End` calls happen inside each addon's Execute() method. The practical impact:
- With Spinner attached: spinner frames may interleave but no crash (Spinner uses internal mutex for writes). The `Status.status` race is on a string assignment which is benign on x86/arm64 in practice, though technically undefined. The `-race` detector WILL flag this.
- Without Spinner (NoopLogger): `Status.Start/End` just calls `logger.V(0).Infof(...)` which goes through the Logger interface. The NoopLogger is safe; the real Logger writes to stderr which is OS-atomic for small writes.

The pragmatic solution: **create a per-goroutine ActionContext copy with a private no-op Status** for Wave 1 parallel execution. This avoids racing on the shared Status. Each goroutine gets its own ActionContext copy with `nodesOnce` still shared (it's a function value pointing to the same sync.OnceValues closure).

Add a helper function:
```go
// parallelActionContext returns a copy of ac suitable for parallel goroutine use.
// It replaces the shared Status with a per-goroutine no-op status to avoid
// racing on Status.status (which is not concurrent-safe). The nodesOnce
// cache (sync.OnceValues) is shared safely across goroutines.
func parallelActionContext(ac *actions.ActionContext) *actions.ActionContext {
    return actions.NewActionContext(
        ac.Context,
        ac.Logger,
        cli.StatusForLogger(log.NoopLogger{}),
        ac.Provider,
        ac.Config,
    )
}
```

Wait -- this creates a NEW sync.OnceValues closure inside NewActionContext. That defeats the purpose: each goroutine would have its own cache. We need to share the nodesOnce.

Better approach: Since `nodesOnce` is unexported on `ActionContext`, we cannot set it from create.go. Instead, we can use a different strategy: call `actionsContext.Nodes()` ONCE before the parallel loop (pre-warm the cache), then each goroutine uses the shared `actionsContext` pointer safely because:
- `nodesOnce` is already resolved (sync.OnceValues returns cached result instantly)
- `Logger` calls are to stderr (atomic for small writes)
- `Status.Start/End` is the only problem

Actually, the simplest correct approach: **pre-warm the nodes cache, then create per-goroutine ActionContexts with no-op Status**. Each gets its own sync.OnceValues, but since the nodes are the same (same provider, same cluster name), this is correct. The only cost is potentially calling ListNodes multiple times, but with pre-warming the original context first and the goroutine contexts being independent, each would call ListNodes once on first use. This is acceptable since ListNodes is cheap (list containers).

Actually, even simpler: the `Status.Start("Installing X")` pattern is cosmetic. During parallel execution, interleaved spinner output is confusing anyway. The log lines (`logger.V(0).Infof(...)`) already provide adequate progress. We should suppress the spinner during parallel execution.

**Final approach:** Create each goroutine's ActionContext with `cli.StatusForLogger(log.NoopLogger{})` which produces a Status with no spinner and a NoopLogger that discards all Start/End output. The `actionsContext.Logger` (the real logger) is passed separately to `runAddonTimed` for log output. But wait, Execute() calls `ctx.Status.Start(...)` internally within each addon. We need the goroutine's ActionContext to have a no-op Status but the REAL Logger.

Revised helper:
```go
func parallelActionContext(ac *actions.ActionContext) *actions.ActionContext {
    // Use the real logger for log output but a no-op status to avoid
    // racing on the shared Status.status field.
    return actions.NewActionContext(
        ac.Context,
        ac.Logger,
        cli.StatusForLogger(log.NoopLogger{}),
        ac.Provider,
        ac.Config,
    )
}
```

Each goroutine calls:
```go
g.Go(func() error {
    pCtx := parallelActionContext(actionsContext)
    res := runAddonTimed(pCtx, addon.Name, addon.Enabled, addon.Action)
    mu.Lock()
    wave1Results[i] = res
    mu.Unlock()
    return nil
})
```

This is safe because:
- Each goroutine has its own ActionContext and Status (no sharing of mutable state)
- Each goroutine has its own nodesOnce (will call ListNodes once per goroutine, but ListNodes is a read-only list operation that is safe to call concurrently on the provider)
- Logger is shared but Logger interface methods write to stderr atomically
- Provider is shared but Provider.ListNodes is read-only

For Wave 2 (sequential), use the original `actionsContext` with the real Status.

Add the `log` import alias if not already present. `log.NoopLogger{}` is from `sigs.k8s.io/kind/pkg/log` which is already imported.
