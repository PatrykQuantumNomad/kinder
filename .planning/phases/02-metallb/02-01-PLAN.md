<objective>
Implement and test the subnet detection and IP pool carving logic for MetalLB.

Purpose: MetalLB needs to know which IP range to use for LoadBalancer services. This range must be auto-detected from the container provider's network (Docker, Podman, or Nerdctl) without user input. The subnet detection parses `network inspect` JSON output, and the pool carving function computes a safe .200-.250 range from the detected CIDR. These are pure functions with well-defined I/O -- ideal for TDD.

Output: `subnet.go` with `detectSubnet`, `parseSubnetFromJSON`, and `carvePoolFromSubnet` functions, plus `subnet_test.go` with comprehensive unit tests.
</objective>

<feature>
  <name>Subnet detection and IP pool carving</name>
  <files>pkg/cluster/internal/create/actions/installmetallb/subnet.go, pkg/cluster/internal/create/actions/installmetallb/subnet_test.go</files>
  <behavior>
    parseSubnetFromJSON:
    - Docker JSON with IPv4+IPv6 IPAM -> returns IPv4 CIDR string (e.g. "172.18.0.0/16")
    - Docker JSON with only IPv6 IPAM -> returns error "no IPv4 subnet found"
    - Podman JSON with subnets array -> returns IPv4 CIDR string (e.g. "10.89.0.0/24")
    - Podman JSON with empty subnets -> returns error
    - Empty JSON array -> returns error "no kind network found"
    - Malformed JSON -> returns error

    carvePoolFromSubnet:
    - "172.18.0.0/16" -> "172.18.255.200-172.18.255.250"
    - "10.89.0.0/24" -> "10.89.0.200-10.89.0.250"
    - "192.168.49.0/24" -> "192.168.49.200-192.168.49.250"
    - "172.19.0.0/16" -> "172.19.255.200-172.19.255.250"
    - "10.0.0.0/20" -> "10.0.15.200-10.0.15.250" (for /17-/24, use broadcast-adjacent block)
    - Invalid CIDR -> returns error
    - IPv6 CIDR -> returns error "only IPv4 subnets supported"

    detectSubnet (integration-level, calls exec.Command on host):
    - providerBinary="docker" -> runs `docker network inspect kind`, parses Docker JSON schema
    - providerBinary="podman" -> runs `podman network inspect kind`, parses Podman JSON schema
    - providerBinary="nerdctl" -> runs `nerdctl network inspect kind`, parses Docker JSON schema
    - Respects KIND_EXPERIMENTAL_DOCKER_NETWORK env for network name override
  </behavior>
  <implementation>
    Create subnet.go in pkg/cluster/internal/create/actions/installmetallb/ with:

    1. Two JSON struct types:
       - dockerNetworkInspect with IPAM.Config[].Subnet for Docker/Nerdctl
       - podmanNetworkInspect with subnets[].subnet for Podman

    2. parseSubnetFromJSON(output []byte, providerName string) (string, error):
       - If providerName == "podman", unmarshal as []podmanNetworkInspect, iterate subnets for IPv4
       - Otherwise (docker/nerdctl), unmarshal as []dockerNetworkInspect, iterate IPAM.Config for IPv4
       - Filter IPv4 using net.ParseCIDR + ip.To4() != nil
       - Return first IPv4 CIDR found, or error if none

    3. carvePoolFromSubnet(cidr string) (string, error):
       - Parse CIDR with net.ParseCIDR
       - Reject IPv6 (ip.To4() == nil)
       - Compute the broadcast address of the network
       - Place pool at broadcast[-51] to broadcast[-1], i.e. .200-.250 for /24
       - For /16: base[0].base[1].255.200-base[0].base[1].255.250
       - For /17-/24: compute last subnet octet, use .200-.250
       - For larger than /16: same pattern on last two octets
       - Return "startIP-endIP" string

    4. detectSubnet(providerName string) (string, error):
       - Determine network name: os.Getenv("KIND_EXPERIMENTAL_DOCKER_NETWORK"), default "kind"
       - Determine binary name from providerName ("docker"/"podman"/"nerdctl")
       - Run exec.Command(binary, "network", "inspect", networkName) on host
       - Capture output via exec.Output
       - Call parseSubnetFromJSON with output and providerName
       - Return parsed subnet

    Use errors.Wrap/errors.New from sigs.k8s.io/kind/pkg/errors (not fmt.Errorf).
    Use exec.Command/exec.Output from sigs.k8s.io/kind/pkg/exec for host-side commands.

    The test file tests parseSubnetFromJSON and carvePoolFromSubnet as pure unit tests (no exec calls needed). detectSubnet is tested only by the integration test path in Plan 02.
  </implementation>
</feature>
