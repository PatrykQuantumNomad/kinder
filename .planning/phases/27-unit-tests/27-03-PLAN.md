  <behavior>
    localregistry Execute() has two execution domains:
    - Host-side: 4 exec.Command calls (registry inspect, run, network inspect, network connect) — NOT interceptable via FakeNode
    - Node-side: 2 calls per node (mkdir + tee hosts.toml) + 1 ConfigMap apply on control plane

    Since host-side calls use the global exec.Command (not node.CommandContext), the full Execute() path requires a real Docker/Podman daemon. Tests that call Execute() directly will either need Docker available or will fail at the host-side step.

    Approach: Test the full Execute() path with a skip guard for environments without Docker. The test verifies:
    - If Docker is available and the host-side steps succeed, the node-side FakeCmd queue is consumed correctly
    - Provider.Info() error propagates correctly (this happens BEFORE host-side calls)
    - No control plane nodes found returns error (this happens AFTER host-side calls, hard to reach without Docker)

    Additionally, test the node-patching logic in isolation by testing internal helper behavior through the Execute flow when Docker is available.
  </behavior>
  <action>
Create `localregistry_test.go` in package `installlocalregistry`:

**Test structure:**

1. `TestExecute_InfoError` — Tests that Provider.Info() failure is propagated. This code path runs BEFORE any host-side exec.Command calls, so it works without Docker.
   - FakeProvider with `InfoErr: errors.New("info failed")` and `InfoResp: nil`
   - Assert Execute() returns error containing "failed to get provider info"
   - This test ALWAYS runs (no Docker needed)

2. `TestExecute_FullPath` — Tests the full Execute() flow. Requires Docker.
   - Guard: `if exec.Command("docker", "version").Run() != nil { t.Skip("requires Docker daemon") }`
   - Create FakeNode with enough FakeCmds for the node-side calls: for 1 control-plane node, need 3 FakeCmds (mkdir, tee, kubectl apply)
   - FakeProvider with `InfoResp: &providers.ProviderInfo{Rootless: false}`
   - The host-side steps will run real Docker commands (inspect/run/network connect) — the test relies on Docker being available
   - After Execute() succeeds, verify node.Calls has at least 3 entries (mkdir, tee for the CP node, and kubectl apply)
   - If Execute() fails on a host-side step, the test should report the error but not fail hard (the point is testing node interactions when the host side works)
   - IMPORTANT: After test, clean up registry container: `exec.Command("docker", "rm", "-f", "kind-registry").Run()` in a defer

3. `TestExecute_NodePatchingErrors` — Tests that node-side errors propagate. Requires Docker (host-side steps must pass first).
   - Guard: same Docker skip as above
   - Table-driven with cases:
     a. "mkdir fails on first node" — FakeCmd{Err: errors.New("mkdir failed")}, errContains "failed to create certs.d dir"
     b. "tee fails on first node" — FakeCmd{} (mkdir ok) + FakeCmd{Err: errors.New("tee failed")}, errContains "failed to write hosts.toml"
   - Same Docker daemon cleanup in defer

IMPORTANT NOTES:
- The `exec.Command` import here is `sigs.k8s.io/kind/pkg/exec` (not `os/exec`). Use `exec.Command("docker", "version")` for the skip guard.
- FakeProvider should NOT implement `fmt.Stringer` — the localregistry code uses a type assertion `ctx.Provider.(fmt.Stringer)` to detect provider name, and if it fails, defaults to "docker". FakeProvider not implementing Stringer means binaryName defaults to "docker", which is correct for the Docker skip guard.
- Each test case MUST create a fresh ActionContext (never reuse — cache prevents reuse).
- Apache 2.0 header, `t.Parallel()` on the Info error test (no Docker side effects), NOT parallel on Docker-dependent tests (they share the kind-registry container).

Alternative approach if Docker-dependent tests are too fragile: Focus entirely on `TestExecute_InfoError` (always works) and add a comment block explaining that node-side patching tests require Docker and are covered by integration testing. This is the HONEST approach per research recommendation. Choose this if the Docker-dependent tests are too complex to stabilize.
