<objective>
Implement the restore-side counterpart to Plan 02: take already-extracted host files (`etcd.snap`, `images.tar`, `pvs.tar`) and re-apply them to a paused cluster's control-plane and worker nodes. This plan is parallel to Plan 02 (no shared files; both depend only on Plan 01) and is composed by Plan 04 inside the orchestrator.

Purpose: Deliver SC2's restore primitives. The cluster MUST be paused before these functions run — Plan 04 handles the lifecycle.Pause / lifecycle.Resume bookends. These three functions are the dangerous part: they mutate node-internal state (etcd data dir, containerd image store, PV tree). Designed for fail-fast with `errors.NewAggregate`.

- `pvsrestore.go` — untar nested per-node tars back to `/opt/local-path-provisioner`.
- TDD coverage with FakeNode for sequencing assertions.
</objective>

  <action>
    Implement etcd restore with HA-safe ordering. Caller (Plan 04) guarantees the cluster is paused (containers stopped). RestoreEtcd internally STARTS the CP node containers (via providerBin start), restores etcd, swaps data dirs, then leaves them ready for `lifecycle.Resume` to bring the rest up.

    HOWEVER per CONTEXT.md "atomicity = pre-flight + fail-fast, no rollback", and to avoid duplicating start logic from Phase 47, RESOLVE BY: caller (Plan 04) does Resume of CP-only first via a private helper, THEN RestoreEtcd assumes CP containers are running. This separation keeps lifecycle ordering in `pkg/internal/lifecycle/`. Document inline that RestoreEtcd preconditions are "CP containers running, etcd static pod about to be displaced."

    Steps inside RestoreEtcd, per CP node:
    1. Move etcd manifest aside so kubelet stops trying to run etcd:
       `node.CommandContext(ctx, "mv", "/etc/kubernetes/manifests/etcd.yaml", "/tmp/etcd-manifest.yaml.kinder-snap").Run()`
    2. Wait briefly (5s) for kubelet to tear down the etcd container.
    3. Copy host file to node:
       `node.CommandContext(ctx, "cp", "/dev/stdin", "/tmp/kinder-restore.snap").SetStdin(hostFile).Run()`
    4. Run etcdctl snapshot restore (etcdctl IS on kindest/node PATH for all supported versions per RESEARCH Open Question 6 — verify during Plan 03 implementation by inspecting the kindest/node Dockerfile or running a probe `which etcdctl`; if probe fails, FALL BACK to a temporary docker container approach as documented):
       ```
       etcdctl snapshot restore /tmp/kinder-restore.snap \
         --data-dir=/var/lib/etcd-restored \
         --name=<node-hostname> \
         --initial-cluster=<cp1>=https://<cp1-ip>:2380,<cp2>=...,<cp3>=... \
         --initial-cluster-token=kind-snap-restore-<unix-ns> \
         --initial-advertise-peer-urls=https://<thisCP-ip>:2380
       ```
       For single-CP: `<name>=https://127.0.0.1:2380`.
       For HA: build the comma-separated list from all CPs' container IPs (use `providerBin inspect --format {{.NetworkSettings.Networks.kind.IPAddress}}`).
    5. Swap data dir: `mv /var/lib/etcd /var/lib/etcd.kinder-old && mv /var/lib/etcd-restored /var/lib/etcd`. On any failure: try to roll back (`mv /var/lib/etcd.kinder-old /var/lib/etcd`) and return error.
    6. Restore manifest: `mv /tmp/etcd-manifest.yaml.kinder-snap /etc/kubernetes/manifests/etcd.yaml`.
    7. Cleanup: `rm -f /tmp/kinder-restore.snap`.

    HA: use `errors.UntilErrorConcurrent` to run steps 1-4 in parallel across CPs (all need the SAME token, so token is computed once outside the loop), then step 5-7 in parallel as well. Document why parallelism is safe: each CP touches only its own node-local etcd data dir.

    RED then GREEN tests (`etcdrestore_test.go`):
    - `TestRestoreEtcdSingleCP_Sequence`: FakeNode for 1 CP; assert command sequence in exact order: mv manifest aside → cp snap → etcdctl snapshot restore → mv data dirs → mv manifest back → rm tmp.
    - `TestRestoreEtcdHA_SameToken`: 3 CPs; assert all 3 etcdctl invocations use the IDENTICAL `--initial-cluster-token` value AND the identical `--initial-cluster` membership string.
    - `TestRestoreEtcdSnapshotRestoreFails_ManifestRestored`: fake etcdctl exits 1 → manifest is moved BACK (rollback semantics for the manifest move only); error returned mentions etcdctl.
    - `TestRestoreEtcdDataSwapFails_OldDataRestored`: fake `mv /var/lib/etcd /var/lib/etcd.kinder-old` succeeds, but `mv /var/lib/etcd-restored /var/lib/etcd` fails → assert rollback `mv /var/lib/etcd.kinder-old /var/lib/etcd` was attempted; error returned.
    - `TestRestoreEtcdContainerIPLookup`: fake `docker inspect` returns "172.18.0.5" → resulting --initial-advertise-peer-urls contains "172.18.0.5:2380".
