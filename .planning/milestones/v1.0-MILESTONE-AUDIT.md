### Integration Issues Found

#### Issue 1 (MEDIUM): All-False Guard in SetDefaultsCluster

**File:** `pkg/internal/apis/config/default.go:95-101`

The all-false guard in internal `SetDefaultsCluster` was designed as a safety net for library callers who bypass v1alpha4 conversion. However, it also fires when a user explicitly sets all 5 addons to `false` in their config YAML. The v1alpha4 conversion correctly produces an all-false internal struct, but `fixupOptions` calls `config.SetDefaultsCluster` which then overrides all fields back to `true`.

**Impact:** A user who writes `addons: { metalLB: false, envoyGateway: false, ... }` (all five disabled) will silently get all addons installed. Single-addon or partial opt-outs work correctly. This is an edge case — most users disable specific addons, not all of them.

**Severity:** Non-blocking (edge case). The common opt-out scenario (disable 1-4 addons) works correctly.

#### Issue 2 (LOW): Integration Script Missing kubectl Context

**File:** `hack/verify-integration.sh`

The script runs `kinder create cluster --name kinder-integration-test` but never sets `kubectl config use-context "kind-kinder-integration-test"` or passes `--context` flags. If the developer has a pre-existing kubectl context active, the script's `kubectl` commands may target the wrong cluster.

**Impact:** False test results if developer has another cluster active. Does not affect the addon code itself.

**Severity:** Non-blocking (test script only, not production code).
